# Dashboard Asset Pipeline

## What It Does

The MQTT Broadcast dashboard is a web-based monitoring interface that requires CSS styling and JavaScript code to run in the browser. The asset pipeline is the system that builds, distributes, and loads these files so the dashboard renders correctly when a user navigates to it.

The package handles asset management automatically — during installation, the required files are published to the host application. When a user visits the dashboard URL, the system locates the correct files and injects them into the page.

## User Journey

1. Developer installs the package via `composer require enzolarosa/mqtt-broadcast`
2. Developer runs `php artisan mqtt-broadcast:install`, which publishes dashboard assets (CSS, JavaScript, manifest) into the application's `public/vendor/mqtt-broadcast/` directory
3. User navigates to the dashboard URL (default: `/mqtt-broadcast`)
4. The server renders the dashboard page, automatically including the correct CSS and JavaScript files
5. The browser loads the files and the React monitoring interface appears
6. When the package is updated, the developer re-publishes assets (`php artisan vendor:publish --tag=mqtt-broadcast-assets --force`) to get the latest dashboard version

## Business Rules

- Assets must be published to the host application before the dashboard can render in production
- The dashboard uses a manifest file to map source code to built output files — if the manifest is missing, the dashboard renders blank with no error message
- Two asset-loading strategies exist: the standard Laravel Vite integration (default) and manual helper methods (`css()`/`js()`) as a fallback
- During development, assets can be served via a live development server with instant reload (Hot Module Replacement) — no publishing needed
- Asset filenames are deterministic (not content-hashed), so republishing after a package update always overwrites the previous version
- Assets are isolated in `vendor/mqtt-broadcast/` to avoid conflicts with the host application's own frontend build

## Edge Cases

- **Assets not published**: the dashboard page loads but appears blank — no visual error is shown to the user. The developer must publish assets manually.
- **Outdated assets after package update**: the dashboard may show old UI or behave unexpectedly if assets are not republished after updating the package via Composer. Running `vendor:publish --tag=mqtt-broadcast-assets --force` resolves this.
- **Development server not running**: if the application is in development mode and the Vite dev server is not started, the browser shows connection errors in the console and the dashboard does not render.
- **Custom public path**: if the host application uses a non-standard public directory, the asset URLs generated by the `asset()` helper may not resolve correctly.
- **Multiple package versions**: publishing assets from different package versions overwrites previous files — there is no versioning or rollback mechanism.

## Permissions & Access

- Asset publishing requires console access (`php artisan`) — it is a developer/deployment action, not available to end users
- The published assets are static files served from the `public/` directory — no authentication is needed to load the CSS and JavaScript files themselves
- Access to the dashboard page (which uses these assets) is controlled by the `viewMqttBroadcast` authorization gate and the `Authorize` middleware — see the Dashboard Monitoring documentation for details
- The `laravel-assets` publish tag allows Laravel 11+ to include MQTT Broadcast assets when running bulk asset publishing (`php artisan vendor:publish --tag=laravel-assets`)
